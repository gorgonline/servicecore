## Spec

### Açık Sorular
1. **Mevcut string stratejisi:** Şu an `AuthShell` ve feature bileşenleri metinleri **prop** olarak alıyor (ör. `eyebrow`, `title`, `trust` default literal). Bu görev i18n'e geçiş diyor ama prop'lı API'nin kaderini netleştirmiyor. **Varsayım:** Bu adım yalnız **tip sözleşmesi + namespace + peerDependency** kurar; bileşenlerin `useTranslation`'a fiilen geçirilmesi (prop→t() migrasyonu) ileri adımlara/feature bileşeni adımlarına bırakılır. Bu spec key tiplerini ve altyapıyı kurar, gövde migrasyonunu zorlamaz.
2. **`AuthShell` markalama metinleri** (SERVICECORE wordmark, tagline) literal/`text-transform` nedeniyle bilinçli sabit — bunlar **i18n dışı** kalır; `ScAuthKeys` yalnız form alanı key'lerini kapsar.
3. **001/002 sırası:** 002 tek-entry barrel'ı kuruyor ve "i18n key tipleri (varsa)" diyor. Bu spec, `src/i18n/keys.ts`'in oluşturulduktan sonra `src/index.ts`'ten re-export edileceğini varsayar (002 ile uyumlu).

### Amaç
`@servicecoreui/ui` paketine **tip-tabanlı i18n sözleşmesi** eklemek. Paket çeviri **metni taşımaz**; yalnız (a) feature bileşenlerinin hangi key'leri beklediğini tanımlayan TypeScript interface'leri (`ScAuthKeys`, `ScLayoutKeys`), (b) `react-i18next`/`i18next` peerDependency'leri, (c) `sc-` namespace konvansiyonu içerir. Çeviri kaynakları **tüketici projede** yaşar; playground tüketici rolünde kendi `locales/tr/sc-auth.ts` + `sc-layout.ts` dosyalarını sağlayıp `addResourceBundle` ile yükler. Eksik key, derleme zamanında (tüketicinin resource objesi `ScAuthKeys`/`ScLayoutKeys`'e `satisfies` ile bağlandığında) yakalanır.

### Kapsam ve Yaklaşım
1. **`src/i18n/keys.ts` oluştur** — iki interface export et:
   - `ScAuthKeys`: `email`, `emailPlaceholder`, `password`, `passwordPlaceholder`, `loginButton`, `rememberMe`, `forgotPassword`, `orContinueWith`, `googleLogin`, `ssoLogin`, `captchaRefresh`, `captchaLabel` + diğer auth formlarının (şifremi unuttum / sıfırla / 2FA / kayıt / yetkisiz) ihtiyaç duyduğu key'ler (ör. `submitButton`, `backToLogin`, `resetPasswordTitle`, `newPassword`, `confirmPassword`, `twoFactorCode`, `resendCode`, `registerButton`, `unauthorizedTitle`, `unauthorizedMessage`). Tüm değerler `string`.
   - `ScLayoutKeys`: `logout`, `notifications`, `search` + gerekli olanlar (ör. `profile`, `settings`, `help`, `menu`, `closeMenu`).
   - Ayrıca namespace sabitlerini export et: `export const SC_AUTH_NS = "sc-auth"` ve `export const SC_LAYOUT_NS = "sc-layout"` (string literal type ile) — tüketici ve bileşen aynı kaynaktan namespace alır, çakışma/yazım hatası önlenir.
2. **`src/index.ts`'ten re-export** — `export type { ScAuthKeys, ScLayoutKeys } from "./i18n/keys";` ve namespace sabitleri (`export { SC_AUTH_NS, SC_LAYOUT_NS } from "./i18n/keys";`). (002 tek-entry ile tutarlı; tek public giriş.)
3. **peerDependency ekle** — `package.json` `peerDependencies`'e `react-i18next` (`>=13.0.0`) + `i18next` (`>=23.0.0`); `peerDependenciesMeta` ile zorunluluğu uygun şekilde işaretle; aynı sürümler `devDependencies`'e (playground/tip çözümü için repo kökünde mevcut değilse) eklenir.
4. **Feature bileşeni kullanım sözleşmesi (dokümante et, gerekirse örnek bileşende uygula)** — feature bileşenleri `useTranslation(SC_AUTH_NS)` / `useTranslation(SC_LAYOUT_NS)` çağırır, `t("loginButton")` gibi `ScAuthKeys` anahtarlarıyla erişir. Bu adımın asgari teslimi tip + namespace + peerDep; bileşen gövde migrasyonu (varsa) bu sözleşmeye uyar.
5. **Playground tüketici kurulumu** — `apps/playground` içinde:
   - `locales/tr/sc-auth.ts` → `const scAuth = { ... } satisfies ScAuthKeys; export default scAuth;` (tüm key'ler dolu; eksik key burada derleme hatası verir).
   - `locales/tr/sc-layout.ts` → `satisfies ScLayoutKeys`.
   - i18next init/bootstrap'ta `i18n.addResourceBundle("tr", SC_AUTH_NS, scAuth)` ve `addResourceBundle("tr", SC_LAYOUT_NS, scLayout)`.
   - `I18nextProvider` (veya mevcut provider) auth/layout demo sayfalarını sarmalar.
